## RBAC System (Role-Based Access Control)

Discord-complexity permission system. Prerequisite for the Moderation System and all future privileged operations.

### Data Model

#### `roomRoles` table

| Column        | Type      | Notes                                                                                  |
| :------------ | :-------- | :------------------------------------------------------------------------------------- |
| `id`          | uuid PK   | defaultRandom()                                                                        |
| `roomId`      | uuid FK   | → `rooms.id` CASCADE DELETE                                                            |
| `name`        | text      | NOT NULL; max 100 chars                                                                |
| `color`       | text      | nullable; hex color string (e.g. `#ff5500`)                                            |
| `position`    | integer   | NOT NULL DEFAULT 0; higher = higher priority in hierarchy                              |
| `permissions` | bigint    | NOT NULL DEFAULT 0; bitfield of `RoomPermission` flags                                 |
| `isEveryone`  | boolean   | NOT NULL DEFAULT false; at most one `true` per room; applied to all members implicitly |
| `createdAt`   | timestamp | metadata                                                                               |
| `updatedAt`   | timestamp | metadata                                                                               |

Index: `(roomId, position DESC)` for hierarchy queries.
Check: `position >= 0`.
Check: at most one `isEveryone = true` per `roomId` (partial unique index).

#### `usersToRoomRoles` table

| Column   | Type    | Notes                           |
| :------- | :------ | :------------------------------ |
| `userId` | text FK | → `users.id` CASCADE DELETE     |
| `roomId` | uuid FK | → `rooms.id` CASCADE DELETE     |
| `roleId` | uuid FK | → `roomRoles.id` CASCADE DELETE |

PK: `(userId, roomId, roleId)`.

> The `@everyone` role has `isEveryone = true`. It is **never** stored in `usersToRoomRoles` — it applies to all members implicitly and is included in every permission query.

### `RoomPermission` Bitfield Enum

Lives in `packages/shared/src/models/room/RoomPermission.ts`.

```typescript
export enum RoomPermission {
  ReadMessages = 1 << 0, // 1     — see message history
  SendMessages = 1 << 1, // 2     — post messages
  ManageMessages = 1 << 2, // 4     — delete/pin others' messages
  MentionEveryone = 1 << 3, // 8     — use @here / @everyone
  ManageRoom = 1 << 4, // 16    — edit room name, image, settings
  ManageRoles = 1 << 5, // 32    — create/edit/delete roles below own top position
  ManageInvites = 1 << 6, // 64    — create/delete invite codes
  KickMembers = 1 << 7, // 128   — remove a member from room (non-punitive)
  BanMembers = 1 << 8, // 256   — permanent ban; creates ban record
  MuteMembers = 1 << 9, // 512   — force-mute/unmute in voice
  MoveMembers = 1 << 10, // 1024  — kick a member from voice channel
  Administrator = 1 << 11, // 2048  — grants ALL permissions; bypasses hierarchy checks
}
```

### Permission Computation

Computed at query time via SQL `BIT_OR`. The owner (`rooms.userId`) always has full permissions and bypasses this entirely.

```sql
-- Effective permissions = @everyone role ∪ all explicitly-assigned roles
SELECT BIT_OR(rr.permissions)::bigint AS permissions
FROM room_roles rr
WHERE rr.room_id = $roomId
  AND (
    rr.is_everyone = TRUE
    OR rr.id IN (
      SELECT role_id FROM users_to_room_roles
      WHERE user_id = $userId AND room_id = $roomId
    )
  )
```

`Administrator` bit set → treat all bits as 1 (bypass all further checks).

### Authority Hierarchy

```
Owner (rooms.userId)          — synthetic; immune to all role manipulation; can do everything
  └── Administrator permission — bypass all checks; can manage all roles below top
        └── Explicit permissions  — granular; subject to role hierarchy
              └── @everyone perms — baseline for all room members
```

### Role Hierarchy Rules

- A user's **top position** = max `position` across their explicitly-assigned roles (not @everyone).
- Owner's top position = `+Infinity` (conceptually).
- A user can only **assign/remove** roles with `position < their own top position`.
- A user can only **edit/delete** roles with `position < their own top position`.
- Exception: `Administrator` permission bypasses hierarchy checks (but still can't touch Owner status).

### Default Role Permissions

| Role        | Default permissions                                                                                                       |
| :---------- | :------------------------------------------------------------------------------------------------------------------------ |
| `@everyone` | `ReadMessages \| SendMessages \| MentionEveryone \| ManageInvites`                                                        |
| `Admin`     | `ManageRoom \| ManageRoles \| ManageInvites \| KickMembers \| BanMembers \| MuteMembers \| MoveMembers \| ManageMessages` |

### Service Layer (`server/services/room/rbac/`)

| Function                                                        | Returns            | Notes                                                                                                        |
| :-------------------------------------------------------------- | :----------------- | :----------------------------------------------------------------------------------------------------------- |
| `getPermission(db, userId, roomId)`                             | `Promise<bigint>`  | SQL `BIT_OR` over @everyone + assigned roles                                                                 |
| `hasPermission(db, userId, roomId, permission: RoomPermission)` | `Promise<boolean>` | Owner bypass → `Administrator` bit → specific bit                                                            |
| `getTopRolePosition(db, userId, roomId)`                        | `Promise<number>`  | Max position across assigned roles (not @everyone)                                                           |
| `isManageable(db, actorId, roomId, targetPosition: number)`     | `Promise<boolean>` | Actor's top > targetPosition. For a user: resolve their top first; for a role: pass `role.position` directly |

### tRPC Procedure Guards

| Guard                                                   | When to use                                                   |
| :------------------------------------------------------ | :------------------------------------------------------------ |
| `getOwnerProcedure(schema, roomIdKey)`                  | Owner-only: `deleteRoom`, `transferOwnership`                 |
| `getPermissionProcedure(permission, schema, roomIdKey)` | Any procedure requiring a specific permission; owner bypasses |

`getCreatorProcedure` is **retired** — replaced by `getOwnerProcedure` and `getPermissionProcedure`.

## Moderation System

Unified admin action system. All procedures gated behind specific `RoomPermission` bits rather than a generic "admin" flag.

- [ ] **`AdminActionType` enum** — `ForceMute | ForceUnmute | KickFromVoice | KickFromRoom | TimeoutUser | BanUser`; lives in `shared/models/message/AdminActionType.ts`
- [ ] **`moderationRouter`** — `server/trpc/routers/moderation.ts`
  - `executeAdminAction({ roomId, targetUserId, type, durationMs? })` — permission gate per action type (see table below); checks `canManageTarget` before execution
  - `onAdminAction({ roomId })` — subscription; targeted `userId` receives the action
  - `readModerationLog({ roomId, cursor })` — cursor-paginated; behind `ManageRoom`
- [ ] **Permission gates per action type**

  | Action          | Required permission |
  | :-------------- | :------------------ |
  | `ForceMute`     | `MuteMembers`       |
  | `ForceUnmute`   | `MuteMembers`       |
  | `KickFromVoice` | `MoveMembers`       |
  | `KickFromRoom`  | `KickMembers`       |
  | `TimeoutUser`   | `KickMembers`       |
  | `BanUser`       | `BanMembers`        |

- [ ] **Force-mute / force-unmute** — targeted client receives `ForceMute`/`ForceUnmute` via `onAdminAction`; sets `isMuted = true` and disables local toggle until `ForceUnmute`
- [ ] **Kick from voice** — targeted client receives `KickFromVoice`; calls `leaveVoice()` locally; shows snackbar
- [ ] **Kick from room** — targeted client receives `KickFromRoom`; navigates away; server deletes `usersToRooms` row
- [ ] **Timeout** — `durationMs` required; server sets `timeoutUntil` column on `usersToRooms`; `createMessage` rejects if `timeoutUntil > now()`
- [ ] **Ban** — permanent; server deletes `usersToRooms` + all `usersToRoomRoles` rows; inserts into `bans` table (`roomId`, `userId`, `bannedByUserId`, `createdAt`); join/invite flows reject banned users; `readBans`/`unbanUser` procedures added behind `BanMembers`
- [ ] **Moderation log** — append-only Azure Table (`AzureTable.ModerationLog`); each row: `{ partitionKey: roomId, rowKey: reverseTickedTimestamp, type, actorId, targetId, durationMs? }`; surfaced under room settings "Audit Log" tab (behind `ManageRoom`)
