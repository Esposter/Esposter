# Esbabbler — v3 Feature Roadmap

All features below are free-to-implement (no paid infra) unless explicitly noted.
Items are grouped by theme; strikethroughs are acknowledged but deferred with rationale.

---

## Completing v2 Open Items

- [ ] **Moderator force-mute** — see Moderation System below; blocked on unified `moderationRouter` design

## Room Member Roles

Prerequisite for the Moderation System. Lightweight, no RBAC engine needed.

- [ ] **`RoomMemberRole` enum** — `Owner | Admin | Member`; added as `role` column on `usersToRooms` (default `Member`; creator of a room gets `Owner` on insert)
- [ ] **`isRoomAdmin` middleware** — reusable tRPC procedure guard that asserts `role IN (Owner, Admin)` for the calling user in the target room; consumed by all moderation procedures
- [ ] **Role assignment UI** — right-click / context menu on a member in the member list; `Owner` can promote/demote to `Admin`; `Admin` can promote `Member` to `Admin` (but not set `Owner`)
- [ ] **`updateMemberRole` procedure** — `room/member.ts`; validates caller's role, updates `usersToRooms.role`; real-time `onMemberRoleUpdate` subscription

> **Note:** No per-category or per-room-channel permission overrides in this pass — a single role per room is sufficient at current scale.

## Moderation System

Unified admin action system designed as a cohesive unit rather than bolted onto existing procedures. Unblocks moderator force-mute from v2.

- [ ] **`AdminActionType` enum** — `ForceMute | ForceUnmute | KickFromVoice | KickFromRoom | TimeoutUser | BanUser`; lives in `shared/models/message/AdminActionType.ts`
- [ ] **`moderationRouter`** — `server/trpc/routers/moderation.ts`; all procedures behind `isRoomAdmin` middleware
  - `executeAdminAction({ roomId, targetUserId, type, durationMs? })` — single entry-point mutation for all action types
  - `onAdminAction({ roomId })` — subscription; all clients in the room listen for their userId; targeted client receives the action and applies it locally
  - `readModerationLog({ roomId, cursor })` — cursor-paginated read of the audit log
- [ ] **Force-mute / force-unmute** — targeted client receives `ForceMute`/`ForceUnmute` via `onAdminAction`; sets `isMuted = true` and disables the local toggle until `ForceUnmute` arrives
- [ ] **Kick from voice** — targeted client receives `KickFromVoice`; calls `leaveVoice()` locally; shows a "You were removed from voice by an admin" snackbar
- [ ] **Kick from room** — targeted client receives `KickFromRoom`; navigates away; server deletes their `usersToRooms` row
- [ ] **Timeout** — `durationMs` required; server sets a `timeoutUntil` column on `usersToRooms`; `createMessage` rejects if `timeoutUntil > now()`; targeted client shows a "You are timed out" state in the composer
- [ ] **Ban** — permanent; server deletes `usersToRooms` row and inserts into a `bans` table (`roomId`, `userId`, `bannedByUserId`, `createdAt`); join and invite flows reject banned users; `readBans`/`unbanUser` procedures added
- [ ] **Moderation log** — append-only Azure Table (`AzureTable.ModerationLog`); each action row: `{ partitionKey: roomId, rowKey: reverseTickedTimestamp, type, actorId, targetId, durationMs? }`; surfaced in room settings under a "Audit Log" tab (admin-only)

## User Profile Editing

v2 added the profile card (view). This pass adds editing.

- [ ] **`updateUser` procedure** — `server/trpc/routers/user.ts`; restricted to `session.userId = targetId`; validates `displayName` (max 32 chars), `bio` (max 160 chars), `avatarUrl`
- [ ] **Avatar upload** — reuse existing Azure Blob SAS upload flow from file attachments; client uploads to a `/avatars/{userId}` blob path; `avatarUrl` stored on the `users` table
- [ ] **Edit profile dialog** — accessible from the bottom-left user panel (click own username/avatar); `displayName`, `bio`, and avatar upload fields; preview shows the profile card as it will appear to others
- [ ] **`bio` column on `users`** — text, nullable, max 160 chars; migration additive; shown in the profile card beneath the display name
