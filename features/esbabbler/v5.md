# Esbabbler — v5 Roadmap

All features are free-to-implement (no paid infra) unless explicitly noted.
Built on existing Azure Table Storage, Postgres, Azure Blob, and Azure Function infrastructure.
Strikethroughs are acknowledged but deferred with rationale.

---

## Comfort gap analysis

After v4 the platform has: messaging, reactions, threads, drafts, voice, DMs, friends, full RBAC, full moderation (timeout/ban/warn/softban/word filter), slash commands, slowmode, read-only channels, system messages, link previews, file uploads, presence, user profiles, global search, push notifications, inbound webhooks, categories, pinned messages.

Core remaining friction points vs Discord: no role mentions, no room identity (icon), no per-room nickname, no unread indicators, no message permalinks, no group DM management, composer missing formatting toolbar.

---

## Quick Wins

1. **Rich text formatting toolbar** — pure client, reuses existing TipTap marks
2. **Collapsible embeds** — client-side localStorage, zero server cost
3. **Room avatar / icon** — Azure Blob SAS reusing existing upload flow
4. **Per-room nickname** — one nullable column on `usersToRooms`

---

## Composer & Message Display

- [ ] **Rich text formatting toolbar** — bold, italic, underline, strikethrough buttons in the composer footer; TipTap `StarterKit` already includes all four marks; add a `RichTextEditor/Custom/FormattingToolbar.vue` with `editor.chain().focus().toggleBold().run()` etc.; purely client-side, no schema changes
- [ ] **Collapsible embeds** — per-message toggle to hide/show link previews and image attachments; state in `localStorage` keyed by message `rowKey`; small chevron on the embed container; no server involvement; mirrors Discord's "hide embed" eye button
- [ ] **Message permalink / deep link** — right-click message context menu → "Copy Message Link"; URL param `?highlight=rowKey`; on room load, if `highlight` present, scroll to + briefly highlight the target row; client-only (`useRoute`, scroll-to logic in `useReadMessages`); no server changes

---

## Room Identity

- [ ] **Room avatar / icon** — add `iconUrl: text | null` column to `rooms` + migration; upload via existing `AzureContainer.PublicUserAssets` SAS flow (reuse `uploadProfileImage` pattern from user profile); displayed in the sidebar room list, room header, and member hover cards; editable in room settings behind `ManageRoom`
- [ ] **Per-room nickname** — add `nickname: text | null` (max 32 chars) to `usersToRooms` + migration; `setNickname({ roomId, nickname })` tRPC mutation (own nick: any member, others' nick: `ManageNicknames` permission — new bit on `RoomPermission`); rendered wherever we display a member's name within that room context (message list, member panel, audit log); falls back to global display name when null

---

## Unread Indicators

- [ ] **Unread room dot** — compare `usersToRooms.lastMessageAt` (already written by slowmode on each send) to `rooms.updatedAt` to determine "has new messages since last visit"; update `lastMessageAt` on room focus/navigation switch (currently only updated on message send); show a filled dot in the sidebar room row when `lastMessageAt < rooms.updatedAt` and room is not currently active; no new column needed — reuses existing fields; no per-message reads; no unread count (Discord shows counts, we skip to avoid read-receipt semantics already deferred twice)

---

## Mentions

- [ ] **Role mentions** — `@RoleName` in composer suggestion list (roles rendered below special mentions, above members); TipTap Mention extension already supports multiple suggestion sources — add a second `MentionType.Role` span type; server: in `getPushSubscriptionsForMessage`, detect role mention spans, query `usersToRoomRoles` for all members of that role, add to the push target set respecting their `notificationType`; permission: none (any member can @role, same as Discord default); store as `<span data-type="mention" data-mention-type="role" data-id="{roleId}" data-label="{roleName}">@{roleName}</span>` — role color applied client-side from `memberMap` store

---

## Group DM Management

- [ ] **Add / remove participants from group DMs** — `addDMParticipant({ roomId, userId })` and `removeDMParticipant({ roomId, userId })` procedures behind `RoomType.DirectMessage` guard; `addDMParticipant` inserts into `usersToRooms` + emits system message "X was added by Y"; `removeDMParticipant` deletes from `usersToRooms` (self-remove = leave); participant list shown in DM header chip row with an "Add People" button (picker from friends list); regenerates `participantKey` on change; blocked users cannot be added
