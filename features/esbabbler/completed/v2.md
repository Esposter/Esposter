# Esbabbler — v2 Feature Roadmap

All features below are free-to-implement (no paid infra) unless explicitly noted.
Items are grouped by theme; strikethroughs are acknowledged but deferred with rationale.

## Completing v1 Open Items

- [x] **Blocked users** — separate `blocks` table (`blockerId`, `blockedId`, composite PK) rather than `FriendshipStatus.Blocked`; block hides user from `searchUsers` in both directions; `sendFriendRequest` rejects if a block row exists in either direction; `blockUser`/`unblockUser`/`readBlockedUsers` procedures added to `friendRouter`
- [x] **Friend request notifications** — Web Push on `sendFriendRequest` via `ProcessFriendRequestNotification` Event Grid event; sends to all receiver push subscriptions; ~~per-user toggle~~ deferred
- ~~**DM unread badges**~~ _(Deferred — read receipts / lastSeenAt tracking is not good UX for this platform.)_
- [x] **Voice UI refactor + DM voice** — move voice from the sidebar to a header button + active-call panel above the message list; add mobile header overflow menu; enable DM voice; add VoiceCall system message; see [`voice-refactor.md`](voice-refactor.md)

## Room Management

- [x] **Room categories** — a `categories` table (`id`, `guildId`, `name`, `position`); rooms gain a nullable `categoryId` FK; sidebar renders collapsible category groups with drag-to-reorder; mirrors Discord's category model but simpler (no per-category permission overrides)
- ~~**Room topic / description**~~ _(Not applicable — Discord doesn't have this either.)_
- [x] **Pinned messages list** — `PinnedMessagesMenuButton.vue` in the room header; cursor-paginated list via `useReadPinnedMessages`, renders via `MessageModelMessageSearchList`
- [x] **Edit own messages** — tRPC `updateMessage` mutation restricted to `authorId = session.id`; edited messages show an "(edited)" label; ~~no edit history~~ (still deferred — store only the latest text, not a version log)
- [x] **Delete own messages** — tRPC `deleteMessage` mutation restricted to `authorId = session.id` (admins can also delete any); renders as a tombstone or removes the row entirely

## Slash Commands

- [x] **`/me [action]`** — parameterized; posts `*action text*` as Markdown italics via regular `MessageType.Message` (no separate MessageType needed)
- [x] **`/flip`** — immediate; client-side coin flip; posts "🪙 Heads" or "🪙 Tails"
- [x] **`/shrug`** — parameterized; optional text prefix + `¯\_(ツ)_/¯`
- [x] **`/tableflip`** — immediate; posts `(╯°□°）╯︵ ┻━┻`
- [x] **`/unflip`** — immediate; posts `┬─┬ノ( º _ ºノ)`

All are pure client-side `createMessage` calls — zero server logic beyond the existing `createMessage` path. Commands with parameters use inline parameter input mode in the composer (Discord-style) with required field validation.

## Voice Channel Improvements

- [x] **Deafen toggle** — client-side only; `setRemoteAudioMuted` in WebRTC store sets `audio.muted` on all remote elements; `isDeafened` in voice store; headphone-slash badge shown on local participant avatar; resets on leave
- [ ] **Moderator force-mute** — room admins can force-mute a participant; `setMute` gains an optional `targetUserId` parameter; `onMuteChanged` carries a `forcedByAdminId` field; targeted client sets `isMuted = true` and disables the local toggle until un-muted by the admin
  > **⚠️ Architecture note (deferred):** Force-mute, kick, and future admin actions (ban, timeout, role assignment) all share the same shape — a privileged caller targeting another participant. Before implementing, design a unified `adminAction` router (or a `moderationRouter`) with a common permission guard (`isRoomAdmin` middleware), a typed `AdminActionType` enum, and a single subscription `onAdminAction` that all clients listen to. Bolting each action onto existing procedures (e.g. extending `setMute` with a `targetUserId`) will create divergent patterns that are hard to audit. Implement this as a cohesive system, not piecemeal.
