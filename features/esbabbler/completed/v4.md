# Esbabbler — v4 Roadmap

All features are free-to-implement (no paid infra) unless explicitly noted.
Items leverage existing Azure Table Storage, Postgres, and Azure Function infrastructure.
Strikethroughs are acknowledged but deferred with rationale.

---

## Quick Wins

Smallest scope, implement first.

1. ✅ **Message drafts** — pure client-side, zero schema changes (see Message Improvements)
2. ✅ **`usersToRooms` partial index** — one migration, no logic changes (see Performance)
3. ✅ **Warn action** — extends existing `AdminActionType` + `moderationRouter`, no new infra (see Moderation)
4. ✅ **`/topic [text]`** — one column + one mutation + header subtitle, well-isolated (see Slash Commands)
5. ✅ **Read-only channels** — one bool column + migration + one guard in `createMessage` (see Room Management)

## Moderation Extensions

Building on v3's `moderationRouter`, `AdminActionType`, and `ModerationLog` Azure Table — no new infrastructure.

- [x] **Warn action** — add `AdminActionType.Warn` with an optional `reason: string` field; write to `ModerationLog` like other actions; `onAdminAction` subscription delivers a snackbar to the target; warn count in audit log computed client-side by counting `AdminActionType.Warn` entries in `readModerationLog` (no new storage); requires `ManageMessages` permission
- [x] **Softban** — `AdminActionType.SoftBan`: kick + mark the user's last N (MAX_READ_LIMIT) Azure Table messages as `deletedAt = now()`; behind `BanMembers` permission; useful for clearing spam without a permanent ban
- [x] **Basic word filter** — per-room blocked-words list stored as `string[]` on a new `roomFilters` Postgres table; `createMessage` checks before persisting and throws `FORBIDDEN` with a user-visible reason; moderators with `ManageMessages` bypass; case-insensitive substring match
- ~~**Auto-prune timed-out messages** — (Removed: `assertNotTimedOut` checks `timeoutUntil > now()` at the app level; stale rows are naturally ignored, no cleanup needed.)~~

## Room Management

- [x] **Read-only / announcement channels** — add `isReadOnly: boolean` (default `false`) to `rooms` Drizzle schema + migration; only applies to `RoomType.Room` (DMs are unconditionally writable); `createMessage` rejects with `FORBIDDEN` if `isReadOnly && !hasPermission(ManageMessages)`; editable in room settings behind `ManageRoom`; shown with a megaphone icon in the sidebar
- [x] **System messages for join / leave** — implemented via single `MessageType.System` (not separate JoinRoom/LeaveRoom types — content is fully described by message text, new type would add no value); post when `usersToRooms` rows are inserted/deleted + joinRoom/leaveRoom; rendered as compact system row with info icon
- [x] **Slowmode** — add `slowmodeMs: integer | null` to `rooms` + migration; add `lastMessageAt: timestamp | null` to `usersToRooms` + migration; `createMessage` rejects with `TOO_MANY_REQUESTS` if `now() - lastMessageAt < slowmodeMs`; update `lastMessageAt` on successful send; users with `ManageMessages` bypass; configurable in room settings

## Message Improvements

- [x] **Message drafts** — persist draft per room in `localStorage` keyed by `roomId`; restore on room switch; clear on successful send; sidebar room list shows a draft indicator (italic "Draft" suffix) when a room has a pending unsent draft; pure client-side, zero server cost
- [x] **Thread view** — "View Thread" action on Message type; opens Thread panel in right drawer (`RightDrawer.Thread`); server: `readThread({ rootRowKey, roomId })` fetches root via `getEntity` + replies via `replyRowKey eq rootRowKey` filter; no new storage

## Slash Commands

Building on the existing slash-command infrastructure in `shared/models/message/slashCommands/`.

- [x] **`/topic [text]`** — sets the room topic (a new `topic: string | null` column on `rooms`); `updateRoom` tRPC mutation behind `ManageRoom`; displayed in a subtitle below the room name in the header; calling `/topic` with no argument clears the topic
- [x] **`/remind [duration] [text]`** — `parseDuration` parses `1h30m`/`10m`/`30s` format; client `setTimeout` fires `new Notification(...)` if `Notification.permission === "granted"`; purely client-side, no server needed
