## File Upload Architecture Fix

### Remove `octetInputParser` — migrate to SAS pattern

**Problem**: `octetInputParser` makes the entire tRPC request body the raw binary stream. This means:

- No room for any other input (e.g. `roomId`) alongside the file — userId has to come from session only, making it impossible to scope uploads to a room
- Size validation is client-side only (`requireAtMostMaxFileSize` in `formRules.ts`) because tRPC treats it as a normal request body sharing `MAX_REQUEST_SIZE`; the `@TODO` comment in `formRules.ts` acknowledges this as a workaround
- `MAX_REQUEST_SIZE` (2 MB) is shared with all tRPC requests — not a dedicated file limit

**Solution**: follow the same two-step SAS pattern already used for message file uploads (`generateUploadFileSasEntities` → `uploadBlocks`). See architecture.md § File Uploads for the pattern reference.

**User router** — `generateProfileImageUploadUrl()`:

- No input; blob path scoped to `${ctx.getSessionPayload.user.id}/ProfileImage` in `AzureContainer.PublicUserAssets`
- Returns SAS write URL; client calls `uploadBlocks(file, sasUrl)` directly to Azure Blob
- Public blob URL is deterministic — `updateUser({ image: blobPublicUrl })` saves it as before
- `MAX_FILE_REQUEST_SIZE` (10 MB) as the SAS max blob size; Azure Blob enforces server-side
- Client: `User/ProfileCard/Column/Image.vue` — replace `$trpc.user.uploadProfileImage.mutate(file)` with two-step flow
- Remove `uploadProfileImage` + `octetInputParser` from `user.ts`

**Tests**: `generateProfileImageUploadUrl` is a normal tRPC mutation returning a URL string — fully testable via `createCallerFactory` + `MockContainerDatabase` (same pattern as `generateUploadFileSasEntities` in `message/index.test.ts`). Add to `user.test.ts`.

## Room Identity

### Room profile image

`image text NOT NULL DEFAULT ""` on `rooms` (column exists but currently nullable — migration needed to add NOT NULL + DEFAULT '').

**Room router** — `generateProfileImageUploadUrl({ roomId })`:

- Checks `hasPermission(ManageRoom)`; blob path `rooms/${roomId}/ProfileImage` in `AzureContainer.PublicUserAssets`
- Returns SAS write URL; client calls `uploadBlocks(file, sasUrl)` directly
- `updateRoom({ roomId, image: blobPublicUrl })` saves the deterministic URL
- Same `MAX_FILE_REQUEST_SIZE` SAS limit as user profile image

Both procedures named `generateProfileImageUploadUrl` — disambiguated by their router (`user.` vs `room.`), matching the implicit context pattern of the rest of the routers.

**Display**: sidebar room list, room header, member hover cards. Empty string = no image (avatar initial fallback).

**Settings**: room settings panel behind `ManageRoom` permission.

**Tests**: same pattern as user variant — `createCallerFactory` + `MockContainerDatabase` + permission guard check. Add to `room/index.test.ts`.
