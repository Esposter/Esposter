# Esbabbler - v7 Roadmap

Goal: make Esbabbler feel more like a durable, polished Discord-style community app without adding expensive infrastructure or undoing the simple storage split that is already working.

All items should be free-to-implement or near-free at current scale. New Azure usage must either fit an existing free grant / free tier, reuse services we already provision, or be gated by the existing budget shutdown strategy.

---

## Cost Guardrails

Use existing services first:

- **Postgres** for relational membership, preferences, events, follows, and idempotency rows.
- **Azure Table Storage** for high-volume append-heavy records: messages, moderation logs, lightweight activity logs.
- **Azure Blob Storage** for files and generated media derivatives.
- **Azure Functions + Event Grid / Queue Storage** for background jobs that can retry independently of the app request.
- **Azure Web PubSub** only where cross-process fan-out is needed; keep tRPC subscriptions as the app-facing API.
- **Azure AI Search** only for message search index improvements; do not introduce semantic/vector search until there is a clear product need.
- **LiveKit** for media; avoid recording/egress features unless pricing is explicitly reviewed.

Pricing references to re-check before implementation:

- [Azure Functions pricing](https://azure.microsoft.com/en-us/pricing/details/functions/) - consumption includes a monthly free grant for executions and GB-s.
- [Azure Web PubSub pricing](https://azure.microsoft.com/en-us/pricing/details/web-pubsub/) - free tier exists; Standard includes a free message allowance before message overage.
- [Azure Queue Storage pricing](https://azure.microsoft.com/en-us/pricing/details/storage/queues/) - cheap per-operation storage primitive for delayed/retried work.
- [Azure AI Search pricing](https://azure.microsoft.com/en-us/pricing/details/search/) - free tier is a sandbox; paid tiers are hourly, so keep budget alarms and shutdown paths.

## Architecture Refactors

### Cross-process event bridge

Current tRPC subscriptions are driven by in-process EventEmitters. That is fine for a single Node process, but it becomes fragile if Railway uses multiple replicas or restarts during active calls.

- [ ] Add a narrow `publishMessageEvent` / `publishRoomEvent` service boundary around the existing EventEmitter calls.
- [ ] Keep local EventEmitter delivery as the immediate fast path.
- [ ] Mirror only cross-process-worthy events to Azure Web PubSub: message create/update/delete, room update, role update, user-to-room update, direct-message participant changes, and call participant state.
- [ ] Add a `server/services/message/events/receiveWebPubSubEvent.ts` dispatcher that re-emits remote events into the local EventEmitter.
- [ ] Include an idempotency key on mirrored events so reconnect/catch-up cannot double-apply store mutations.
- [ ] Keep tRPC subscription procedure shapes unchanged.

Why now: v7 adds more real-time surfaces. Putting a boundary in place before adding scheduled events, presence improvements, and richer call state keeps the architecture calm.

### Subscription/store cleanup pass

- [ ] Audit message, room, user-to-room, emoji, friend, and call subscriptions for duplicated caller-side store mutations.
- [ ] Prefer the established "subscriptions are source of truth" rule; keep optimistic local state only for message send and file upload progress.
- [ ] Add small tests around stores that currently mutate both in mutation wrappers and subscription handlers.
- [x] Standardize subscription watch sources to stable primitive keys, following the direct-message room-id string pattern.

### Search index ownership

The infra already provisions Azure Search in production. Make the app-side ownership clearer.

- [ ] Document the exact `messages-index` schema in `features/esbabbler/architecture.md`.
- [ ] Add an admin-only `readSearchIndexStatus` procedure that reports last indexed message timestamp, indexer status when configured, and stale-index warnings.
- [ ] Add a rebuild/run-indexer action only for owners/admins in development or protected admin contexts.
- [ ] Keep the existing Azure Table source of truth; Search remains a read model.

## Reliable Reminders And Scheduled Work

This revisits the v4 deferred `/remind` Queue idea now that the project explicitly allows cheap Azure services.

### Data model

Postgres table: `scheduledMessageJobsInMessage`

| Field           | Notes                                                 |
| --------------- | ----------------------------------------------------- |
| `id`            | UUID primary key                                      |
| `userId`        | creator / recipient                                   |
| `roomId`        | nullable; null for personal reminder                  |
| `messageRowKey` | nullable source message for "remind me about this"    |
| `type`          | `Reminder`, `ScheduledMessage`, later `EventReminder` |
| `payload`       | JSON text under a strict Zod schema                   |
| `runAt`         | timestamp                                             |
| `completedAt`   | nullable                                              |
| `cancelledAt`   | nullable                                              |

### API

- [ ] `message.scheduleReminder({ roomId?, messageRowKey?, runAt, text })`
- [ ] `message.scheduleMessage({ roomId, runAt, message })` behind normal `SendMessages` + read-only/slowmode checks at creation time and again at execution time.
- [ ] `message.cancelScheduledJob({ id })`
- [ ] `message.readScheduledJobs({ roomId? })`

### Worker flow

1. tRPC mutation writes the Postgres job row.
2. If the delay is short enough for Queue visibility timeout, enqueue directly.
3. If it is long-running, an Azure Function timer periodically enqueues due jobs.
4. Function claims the row transactionally, performs the action, marks `completedAt`.
5. Failure leaves the row incomplete and lets retry/backoff pick it up.

Client UX:

- `/remind 2h check this` uses the reliable server path when signed in.
- Message action: "Remind Me" opens a compact menu of durations.
- Composer schedule button lets a user schedule a future message.
- Scheduled jobs panel in the user settings area.

## Community Events

Discord-style scheduled events are useful and fit the existing architecture well.

### Data model

Postgres table: `roomEventsInMessage`

| Field           | Notes                                                |
| --------------- | ---------------------------------------------------- |
| `id`            | UUID primary key                                     |
| `roomId`        | owning room                                          |
| `userId`        | creator                                              |
| `title`         | normalized string                                    |
| `description`   | nullable text                                        |
| `startsAt`      | timestamp                                            |
| `endsAt`        | nullable timestamp                                   |
| `callSessionId` | nullable; pre-created standalone or room call target |
| `cancelledAt`   | nullable                                             |

Postgres table: `roomEventResponsesInMessage`

| Field      | Notes                             |
| ---------- | --------------------------------- |
| `eventId`  | FK                                |
| `userId`   | FK                                |
| `response` | `Interested`, `Going`, `NotGoing` |

### API and permissions

- [ ] Add `RoomPermission.ManageEvents` before `Administrator`; migration must shift the Administrator bit and update stored role values.
- [ ] `roomEvent.createRoomEvent`, `updateRoomEvent`, `deleteRoomEvent` behind `ManageEvents`.
- [ ] `roomEvent.readRoomEvents({ roomId })` for members.
- [ ] `roomEvent.updateRoomEventResponse({ eventId, response })` for members.
- [ ] Optional reminders use the scheduled-job worker above.

Client UX:

- Event pill in room header when an event is upcoming.
- Events tab in room settings for management.
- Event list drawer from the header action buttons.
- "Join Call" appears when the event is live and linked to a call session.

- [ ] **Automod actions** - extend `roomFiltersInMessage` with action type: reject, warn, timeout. Keep substring matching first; no AI moderation service in v7.

- [ ] **Audit log filters** - filter by actor, target, action type, and date range over `ModerationLog`.
- [ ] **Softban preview** - show how many messages would be deleted before confirming.

Azure note: automod should run in the request path for messages and in Azure Functions only for async cleanup. Do not add a paid AI moderation dependency unless there is a clear abuse problem.
