# File Table Editor — Feature Roadmap v3

## Computed Columns — Extensions

These are out of scope for the v2 computed columns implementation (see `computed-columns-architecture.md`) but natural next steps once v2 ships.

- [ ] **Expression transformations** — multi-column formulas like `{Price} * {Quantity}`; requires a formula parser and a `sourceColumnIds: string[]` (plural) instead of the single `sourceColumnId`. The `resolveValue` resolver would need to pass the full row so the evaluator can pull multiple source values.

- [ ] **Chained computed columns** — allow a computed column to source another computed column (currently blocked by the "no chaining" constraint). Requires cycle detection at definition time (DFS on the column dependency graph) and a topological evaluation order.

- [ ] **Aggregation transformations** — column-level aggregates referencing the whole dataset (e.g. `% of total`, `rank`, `running sum`). Breaks the row-local evaluation model; needs a second evaluator that receives `(rows, column, rowIndex)`.

## Other Column Enhancements

- [ ] **Column-level format strings** — display formatting independent of the stored value (e.g. `$#,##0.00` for numbers, `DD/MM/YYYY` for dates). Pure presentation layer; no change to `row.data`.
