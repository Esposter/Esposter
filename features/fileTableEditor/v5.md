# File Table Editor — Feature Roadmap v5

## Column Enhancements

- [ ] **Configurable footer aggregate** ⚡ — add a `footerStat?: FooterStatType` field to `NumberColumn` (enum: `Sum | Average | Min | Max | Count | None`, default `Sum`). `FooterSlot.vue` looks up the chosen stat via `ColumnStatDefinitionMap` instead of hardcoding Σ. The per-column choice is set in the column edit form via a select.
- [ ] **Number format presets** — extend the column edit form with a "Presets" quick-select that pre-fills `NumberFormat` fields (e.g. Currency USD, Percentage, Thousands, Scientific). No new model fields needed — just UI shortcuts that write into the existing `format` object.

## Visualization

- [ ] **String / Date column charts** ⚡ — `computeColumnChartData` returns `null` for String and Date columns, so the chart button in `StatsDialog` is hidden for them. Add:
  - String → horizontal bar chart of top-10 most frequent values (data from `mostFrequentValue`-style count map)
  - Date → monthly histogram (group by `YYYY-MM`, count rows per bucket)

  Enables the chart button in `ActionSlot.vue` for those types.

- [ ] **Computed column stats** ⚡ — `ColumnStatDefinitionMap` `applicableColumnTypes` never includes `ColumnType.Computed`, so the stats dialog shows a blank row for every computed column. Add a `getComputedColumnEffectiveType(column: ComputedColumn): ColumnType` helper that maps each `ColumnTransformationType` to its output type (Math → Number, String → String, DatePart → Number, etc.). Use it in `computeColumnStats` to substitute the effective type when building the stat context.

## Export / Import

- [ ] **Export as any format** ⚡ — the export dialog title says "Export as {dataSourceType}" and only offers the source format. Expose a format selector (`DataSourceTypeItemCategoryDefinitions`) at the top of `ExportDialog.vue` so users can export CSV data as JSON/XLSX and vice versa. All serializers exist in `DataSourceConfigurationMap`; the dialog already wires `filterDataSourceColumns` + `filterDataSourceRows`.
- [ ] **Export selected rows only** ⚡ — when `selectedRowIds` is non-empty, add an "Export selected rows only" checkbox to `ExportDialog.vue`. Filter `filteredRows` to the selected subset before passing to the serializer. Mirrors how column selection already works in the same dialog.

## UX / Workflow

- [ ] **Column visibility quick menu** ⚡ — add a "Columns" icon button to `Row/TopSlot.vue` (or the table toolbar area) that opens a `v-menu` with a checkbox list of all columns. Calls existing `useToggleColumnVisibility` per item. Eliminates the need to switch to the Column tab just to hide a column.
- [ ] **Keyboard cell navigation** ⚡ — in `ItemSlot.vue`, extend the existing inline-edit keyboard handlers:
  - `Tab` / `Shift+Tab` — commit current edit and move focus to next / previous editable cell in the same row
  - `ArrowDown` / `ArrowUp` — commit and move to the same column in the row below / above
  - `Escape` — already implemented (cancel)
  - `Enter` — already implemented (commit)

  Requires a shared "which cell is focused" signal (a small store ref or an emitted event bubbling up through the row).

- [ ] **Column reorder by dragging row headers** — add drag handles to the `<thead>` column headers in `Row/Table.vue` using `VueDraggable` so columns can be reordered without switching to the Column tab. Calls existing `useReorderColumns`. Disable drag when search or sort is active (mirrors the existing Column/Table behaviour).
- [ ] **Named checkpoints** — extend `useFileHistoryStore` with a `checkpoints: Checkpoint[]` list (name, snapshot index). A "Save checkpoint" button in the undo toolbar lets users name the current position; a "Restore" menu lets them jump back to any named checkpoint. Builds directly on the existing history stack — a checkpoint is just a labeled pointer into it.

## Data Quality

- [ ] **Row annotation** ⚡ — add optional `notes?: string` to the `Row` model (schema + class). Show a small note icon in `ActionSlot.vue` (filled when notes exist, outlined when empty); clicking opens a single-field popover to view/edit. Annotations survive undo/redo via the existing command system.
- [ ] **Computed column value preview** — when editing a Computed column in `EditDialogButton.vue`, render a read-only "Preview" table showing the first 5–10 computed values (column name + computed result) pulled from `editedItem.dataSource.rows` + `computeValue`. Updates live as the user changes the transformation, giving instant feedback before saving.

## Refactoring

### Component Architecture

- [ ] **`FooterSlot.vue` — replace inline sum with `ColumnStatDefinitionMap`** ⚡ — the footer hardcodes a Σ sum (`reduce` over `computeValue`) instead of using the existing stat infrastructure. Add a `sum` entry to `ColumnStatDefinitionMap` (`applicableColumnTypes: [ColumnType.Number]`) and use it in the footer. Also unblocks the "configurable footer aggregate" feature above.
- [ ] **`StatsDialog.vue` — extract `computeColumnStats` to a composable** ⚡ — `computeColumnStats(editedItem.value.dataSource)` runs inside a component `computed` that fires on every `editedItem` deep-change. For large datasets this is O(rows × columns × stats). Extract to a `useColumnStats()` composable so it can be memoized per `dataSource` identity and shared between `StatsDialog` and future consumers.
- [ ] **`computeColumnChartData` — replace if/else with `ColumnChartDataMap`** ⚡ — the current imperative `if (Number) ... if (Boolean) ... return null` diverges from the consistent `*Map` pattern used everywhere else (`ColumnTypeColorMap`, `ColumnTypeCommandMap`, `ColumnStatDefinitionMap`). Replace with `const ColumnChartDataMap: Partial<Record<ColumnType, (stats: ColumnStats) => ColumnChartData | null>>` and a single lookup. Makes adding String/Date charts (see Visualization above) a one-liner.

### Type Safety

- [ ] **`computeColumnStats` — remove `as ColumnStats` cast** ⚡ — `{ columnName, columnType, ...Object.fromEntries([...ColumnStatDefinitions].map(...)) } as ColumnStats` is unchecked; TypeScript cannot verify the spread is complete. Restructure to build the result with explicit typed field assignments or a typed accumulator that satisfies `ColumnStats` without casting.
- [ ] **`ColumnTransformationType.String` vs `StringSplit` — document or align** — the v4 spec described `Split` as `StringTransformationType.Split` inside `ColumnTransformationType.String`, but the code has two separate top-level entries (`String` and `StringSplit`). Either add a comment in `ColumnTransformationType.ts` explaining the divergence, or consolidate `StringSplit` into `String` as a `stringTransformationType` variant.

### Utilities

- [ ] **Extract `formatNullable` to a shared util** ⚡ — `ColumnStatDefinitionMap.ts` defines `const formatNullable = (value: null | number | string) => (value === null ? "—" : String(value))` as a module-private function. Extract to `app/util/formatNullable.ts` and import where needed.
- [ ] **`emptyPercent` — broaden to all types or rename to `nullPercent`** ⚡ — currently `applicableColumnTypes: [ColumnType.String, ColumnType.Date]` only. Number and Boolean columns also have nulls and the formula is identical. Either apply universally and rename to `nullPercent`, or document why String/Date are the only targets.
- [ ] **`ColumnTypeColorMap` — derive Computed column color from output type** ⚡ — Computed columns have a fixed chip color regardless of transformation output type. Add `getEffectiveColumnColor(column: Column): string` that falls back to the transformation output type's color for `ColumnType.Computed`, using the same `getComputedColumnEffectiveType` helper introduced for stats. Keeps the column table visually consistent.
- [ ] **`getItemId` — document the separator contract** — `getItemId` is a one-liner (`` `${id}:${columnName}` ``) imported across many files. The `:` separator is load-bearing. Add a brief comment confirming that UUIDs and column names cannot contain `:`, or add a runtime assertion if that guarantee could ever be violated.

### Test Coverage

- [ ] **Tests for `useFindReplace`** — no test file exists. Cases: no-op when `findValue === replaceValue`, replaces all matching cells, replaces a specific `{ columnName, rowIndex }` cell only, pushes exactly one command to history, no-op when no matches found.
- [ ] **Tests for `useNullStrategy`** — no test file. Cases: each `NullStrategy` variant (fill with value, remove rows, fill with previous, fill with next), undo/redo round-trip.
- [ ] **Tests for `useStringTransformation`** — no test file. Cases: each `StringTransformationType`, only string columns affected, pushes to history.
