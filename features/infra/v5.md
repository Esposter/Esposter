# Infrastructure v5 Roadmap

Managed identity and non-key authentication review for Azure SDK clients.

## Scope

This phase requires application code changes and Azure RBAC assignments.

Do not disable local/key auth until the corresponding code path is migrated and deployed.

## Work Items

- [ ] Keep Railway-hosted app clients on key-based authentication until a non-Azure managed identity replacement is designed.
- [ ] Migrate blob clients from connection strings only when the download/upload URL strategy preserves existing expiry requirements.
- [ ] Replace storage service SAS generation only if the replacement does not shorten existing link lifetimes or disrupt published survey/message asset flows.
- [ ] Keep Azure Search local authentication enabled while the main app is hosted on Railway and uses `AzureKeyCredential`.
- [ ] Keep main app Event Grid publisher key-based while the main app is hosted on Railway.
- [ ] Keep Web PubSub service clients key-based while the main app and functions share a connection-string client path.
- [ ] Add required Azure Functions RBAC assignments in Pulumi where Azure-hosted functions already use managed identity.
- [ ] Deploy app/function code changes.
- [ ] Disable the corresponding local/key auth settings after smoke testing.
- [ ] Move this file to `features/infra/completed/v5.md`.

## Current Findings

All v5 work requires application code changes before any hardening Pulumi changes. The sequence for each client is:

1. Migrate the app or function code to use managed identity (or user delegation SAS).
2. Add the required Azure RBAC role assignment in Pulumi.
3. Deploy the updated code.
4. Smoke test the affected flow.
5. Disable the corresponding local/key auth setting in Pulumi.
6. Run `pnpm infra:preview` and `pnpm infra:up`.

Current blockers per client are documented in [`security-constraints.md`](../../packages/infra/docs/security-constraints.md). No Pulumi changes should be made ahead of step 3.

The Railway-hosted main app does not have an Azure managed identity. Do not replace main app `AzureKeyCredential` or connection-string clients with `DefaultAzureCredential` unless a concrete Railway-compatible identity mechanism is introduced and smoke-tested.

The Azure Function Apps already have `SystemAssigned` managed identity enabled. Function-side Event Grid publishing already uses `DefaultAzureCredential`.

Pulumi should own all live Azure RBAC assignments in the subscription, even when a grant is not part of the current managed-identity migration. Adopt live role assignments with their existing GUIDs, then remove temporary Pulumi `import` options after the adoption apply succeeds.
