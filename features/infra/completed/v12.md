# Infrastructure v12 Roadmap

Split `src` by provider, then bring GitHub repository settings under Pulumi management.

## Motivation

`packages/infra` is a single Pulumi project that, so far, only manages Azure (`@pulumi/azure-native`). We want to manage GitHub repository settings (branch protection, rulesets, webhooks, labels, environments, collaborators/teams, deploy keys, Actions variables, and — names-only — secrets) in the same project as code.

Adding a second provider makes the flat `src/{resources,constants,services}` layout ambiguous, so we first split `src` by provider:

```
src/
  index.ts            # generated ctix barrel (covers everything) — stays at top level
  index.test.ts
  azure/
    resources/Microsoft.*/...
    constants/
    services/
  github/             # added in Phase 2
    resources/
    constants/
```

A separate Pulumi **package** is unnecessary and excessive: one program can load multiple providers in the same `prod` stack. (Stacks are config instances of a project, not folders.)

## Safety Invariants

- **Moving files does not change Pulumi state.** A resource's URN is derived from its `type` + logical `name` + `parent` + stack — never its file path. Phase 1 must therefore be a `pulumi preview` **no-op** (zero creates/updates/deletes/replaces). If preview shows any change, stop and investigate before applying.
- **`@/` alias target is unchanged.** `@/*` → `src/*` (inherited via `configuration/tsconfig.vue.json` `${configDir}/src/*`). The move only changes import _specifiers_ (`@/constants/X` → `@/azure/constants/X`), not the alias.
- **GitHub secrets are write-only.** The API returns secret _names_ but never values. Existing Actions/Dependabot/webhook secret values cannot be imported — they are re-supplied via Pulumi config/ESC and never committed. Actions _variables_ (non-secret) import with values.

Only the hand-refactored `src/github/` files land in the repo. (No repo `generated/` dir; the `.gitignore` entry was removed.)

  - Dropped `branchProtection-{migrate-assets,nuxt5}` from the import: those branches are protected only via the **ruleset**, not classic rules, so classic import 404s. The ruleset covers them.

  - `repository.ts`, `actionsPermissions.ts`
  - ~~`rulesets/copilotReviewDefaultBranch.ts` (existing 11555604)~~ — imported then dropped as unnecessary (`b8ab2fae9`); only the 2d `rulesets/developMainProtection.ts` remains.
  - `labels/*` (12), `collaborators/*` (3)
  - `environments/develop.ts`, `environments/production.ts` (the **active** Railway envs)
  - `secrets/*` (5) — `plaintextValue` sourced from Pulumi config/ESC

**Decisions (locked):**

- **Secrets** → manage via ESC/config. Code reads each value from config (`config.requireSecret(...)`); the real values must be set in ESC before `up`. Until set, preview/up errors on missing config (by design — no placeholder that could overwrite a live secret).
- **Environments** → all 4 are Railway-created (`railway-app[bot]`). The parens-style pair (`Esposter (Esposter / develop|production)`) is **stale** (last deploy 2025-12-09, old Railway naming) → **prune** (delete from GitHub). The non-parens pair is **active** (deploys today) → keep, managed in code.
- **Branch protection → rulesets.** Rulesets are GitHub's modern replacement; migrate `develop` + `main` classic rules into **one** ruleset and retire the classic rules. Note classic per-user force-push bypassers do not carry over — rulesets bypass by role/team/app only.

### 2d. Branch-protection → ruleset migration + the original ask

`src/github/rulesets/developMainProtection.ts` — one ruleset, target `develop` + `main`:

- `pull_request` rule with `required_approving_review_count: 0` ← **the original ask** (no human review required; CodeRabbit stays advisory — GitHub can't gate merge on a bot review).
- `required_status_checks` (strict), `non_fast_forward`, `deletion`.
- Bypass actors: **Admin repository role + the Renovate GitHub App**.

**Safe two-step apply (no protection gap on `main`):**

The classic `branchProtection-{develop,main}` resources live in Pulumi **state** (imported in 2c) but are absent from **code**. A naive first `up` would therefore plan to _delete_ them — which both breaks "additive only" and trips their `protect: true` guard. Keep the legacy resources in code until the new ruleset is live, so the plans match the intended sequence:

1. [x] Author the classic `branchProtection-{develop,main}` resources in code (mirroring the imported names so state lines up). `pnpm infra:up` #1 — _additive only_: creates the new ruleset; classic protections stay untouched (present in both state and code). `main`/`develop` now covered by both classic + ruleset.
2. [x] `pulumi state unprotect` the classic `branchProtection-{develop,main}` and the 2 stale environments; remove all four from code; `pnpm infra:up` #2 — deletes classic protections + stale envs from GitHub. Final state: only `rulesets/developMainProtection.ts` + the 2 active envs remain in code; no classic `branchProtection/` files.
3. [x] Final `pnpm infra:preview` — **no-op confirmed: 122 unchanged**, zero create/update/delete/replace (`update 399`).

## Rollback

- Phase 1 is a pure refactor; `git revert` restores the prior layout with no infra effect.
- Phase 2 imports are non-destructive (import only reads state into Pulumi). The only mutating step is 2d; revert the resource change and `infra:up` to restore required reviews.
