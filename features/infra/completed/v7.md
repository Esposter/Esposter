# Infrastructure v7 Roadmap

Naming migration orchestration and first low-risk wave.

## Scope

- Verify Pulumi owns every live Azure resource before any rename wave.
- Establish the rename order from low operational risk to high operational risk.
- Prepare the first wave without changing stateful service endpoints or increasing steady-state cost.

## Guardrails

- Run `az resource list` for both resource groups before every wave and import anything missing before replacing parent resources.
- Run `pnpm infra:preview` before every apply and stop if the preview contains an unexpected delete, replacement, SKU change, or cost-increasing parallel resource.
- Keep target SKUs, capacity, retention, public access, network rules, and auth modes identical to the legacy resource unless a separate optimization task approves a change.
- Keep old and new resources side by side only for the shortest cutover window needed by that resource type.
- Keep `protect: true` on both legacy and target resources until cutover is verified.
- Update app and Azure Functions environment values in the same phase as the endpoint they reference.

## Current Ownership Check

All currently known live resources in the development and production resource groups are represented in Pulumi, including App Service plans, Smart Detector alert rules, Operations Management solutions, role assignments, storage child resources, and Event Grid subscriptions.

Before starting each later phase, repeat the live inventory check because Azure can create dependent resources outside Pulumi during normal portal or platform operations.

## Do Not Rename Independently

These resources should move only with the parent resource that creates or owns the operational context:

| Resource                        | Reason                                                                                                                            |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| App Service plans               | Current plans are Consumption/Dynamic `Y1`; recreating them is only useful when the Function App moves.                           |
| Smart Detector alert rules      | Names intentionally include the monitored Application Insights component; recreate with the target Application Insights resource. |
| Operations Management solutions | Tied to the Log Analytics workspace solution identity; recreate with the target workspace.                                        |
| Role assignments                | Azure names are GUIDs. Keep readable Pulumi file/export names, but do not rename the Azure GUID resource for aesthetics.          |
| Storage child resources         | Blob service properties and management policies should move with the target storage account.                                      |

The first resource-changing wave is v8. It should begin by creating target-named development API connections, Logic Apps, action groups, and any event subscriptions that can be safely paused or swapped without duplicate delivery. Do not delete the legacy resources in the same apply.
