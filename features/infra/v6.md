## Current Findings

The current Pulumi stack is named `dev` and contains all imported resources from both the `d-shp-rg-esposter-auea-001` (development) and `p-shp-rg-esposter-auea-001` (production) resource groups. The stack name does not reflect that it owns production resources.

Resource names follow the legacy `<env>-shp-<type>-esposter-auea-<idx>` convention with `d` and `p` prefixes to separate environments within the single stack. Compact resources such as storage accounts and search services remove hyphens, for example `pshpstespauea001`.

The legacy convention has several readability issues:

- `shp` is redundant because every imported resource uses the shared-platform scope.
- `d` and `p` are less readable than `dev` and `prod`.
- resource type appears after environment/scope, which makes Azure resource lists harder to scan.
- `evgts` and `pubsub` do not match current Microsoft Cloud Adoption Framework abbreviations; target names should use `evgs` for Event Grid subscriptions and `wps` for Web PubSub.

All existing Azure resources are imported. Renaming most Azure resources requires a replace operation (destroy + recreate), which is high-risk for protected production resources.

Splitting into separate stacks would require importing production resources into a new stack and removing them from `dev`, with careful state management to avoid accidental deletes.

## Decisions

- Keep one unified stack; no environment split.
- Rename stack from `dev` to `prod`.
- Adopt the target naming convention in `packages/infra/docs/naming-conventions.md` for new resources and intentional replacements.
- Keep existing legacy Azure resource names until each rename is planned as explicit migration work.

Target shape:

```text
<environment>-<assetType>-<workload>-<region>-<index>
```

Compact shape for resources that do not allow hyphens:

```text
<environment><assetType><workload><index>
```

Global resources that allow hyphens omit the region unless a second instance in another Azure region is planned:

```text
<environment>-<assetType>-<workload>-<index>
```

The target convention orders components by scan priority: environment first, resource type second, workload third, then region and index. It removes the `shp` scope token, uses readable environment names, keeps compact region tokens such as `auea`, and aligns resource-type abbreviations with Microsoft Cloud Adoption Framework recommendations when available.

## Migration Rules

- Treat each existing resource rename as a separate replacement/migration item with its own preview and rollback notes.
- Do not rename state-only Pulumi logical names separately from Azure names unless the change improves source readability without changing live infrastructure.
- Prefer adopting the target name only when a resource is already being recreated for another reason.
- For globally named resources with public endpoints, plan DNS/client configuration cutover before replacement.
- For storage accounts, plan data copy, connection-string/SAS updates, Event Grid subscriptions, role assignments, lifecycle policy, blob service properties, and application configuration changes before replacement.
- For role assignments, keep the human-readable Pulumi logical name pattern; the Azure resource name remains the live assignment GUID.

## Rename Execution Notes

Azure does not support an in-place rename for the managed resource types in this stack. The migration is a replacement workflow:

1. Create target-named resources alongside the legacy resources.
2. Copy stateful data and recreate dependent child resources.
3. Update app, Azure Function, Logic App, Event Grid, Web PubSub, Search, and storage configuration.
4. Verify traffic and background processing against the target resources.
5. Remove legacy resources from Pulumi state/Azure only after cutover is verified.

Current live resource groups also contain Azure-created resources that are not currently represented in Pulumi source:

| Live Resource Type             | Examples                                                           | Migration Approach                                                                                                                                                                        |
| ------------------------------ | ------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| App Service plan               | `ASP-dshprgesposterauea001-94fb`, `ASP-pshprgesposterauea001-b082` | Adopt into Pulumi as `azure_native.web.AppServicePlan`. Do not replace just for naming; both current plans are Consumption/Dynamic `Y1`, and adoption is cost-neutral.                    |
| Smart detector alert rules     | `Failure Anomalies - ...`, `Potential Memory Leak - ...`           | Adopt into Pulumi as `azure_native.alertsmanagement.SmartDetectorAlertRule`. Do not replace just for naming; these are Application Insights generated rules and adoption is cost-neutral. |
| Operations Management solution | `LogicAppsManagement(...)`                                         | Adopt into Pulumi as `azure_native.operationsmanagement.Solution`. Do not replace just for naming; this is tied to the Log Analytics workspace solution and adoption is cost-neutral.     |

Adoption rule: if Azure created a dependent resource and Pulumi has a first-class Azure Native resource type for it, import/adopt the live resource before any parent replacement. Renaming those dependents is lower priority than owning them safely. A target name should be used only when the parent resource is already being recreated for the broader migration.

## App Cutover Inputs

Application code is mostly environment-driven. A resource replacement requires updating these values in the relevant deployment environment:

| Setting                                   | Needs Update When                                  |
| ----------------------------------------- | -------------------------------------------------- |
| `AZURE_CONTAINER_BASE_URL`                | Storage account name changes.                      |
| `AZURE_STORAGE_ACCOUNT_CONNECTION_STRING` | Storage account name/key changes.                  |
| `AZURE_EVENT_GRID_TOPIC_ENDPOINT`         | Event Grid topic name/resource changes.            |
| `AZURE_EVENT_GRID_TOPIC_KEY`              | Event Grid topic is recreated or keys are rotated. |
| `AZURE_FUNCTION_BASE_URL`                 | Function app name changes.                         |
| `AZURE_FUNCTION_KEY`                      | Function app is recreated or keys are rotated.     |
| `AZURE_SEARCH_BASE_URL`                   | Search service name changes.                       |
| `AZURE_SEARCH_API_KEY`                    | Search service is recreated or keys are rotated.   |
| `AZURE_WEB_PUBSUB_CONNECTION_STRING`      | Web PubSub resource name/key changes.              |

## Stack Rename Steps

Run from `packages/infra/`:

```powershell
pulumi stack rename prod
```

Pulumi stack rename updates the backend record only. Resource URNs are based on project name and resource logical names, not the stack name, so no resources are affected. Run `pnpm infra:preview` after the rename to confirm the preview is still clean.
