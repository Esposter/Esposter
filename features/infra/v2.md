# Infrastructure v2 Roadmap

Storage safety and lifecycle review. This phase separates safe storage improvements from migrations that require app changes.

## Scope

- Do not disable shared key access.
- Do not disable account-level blob public access.
- Do not change storage network default action to `Deny`.
- Identify lifecycle, backup, soft-delete, versioning, and redundancy changes that can be applied safely.

## Current Findings

- `dshpstespauea001` and `pshpstespauea001` use `Standard_LRS`.
- Both storage accounts keep `allowSharedKeyAccess: true`; current app blob and SAS flows require shared-key or connection-string auth.
- Both storage accounts keep `allowBlobPublicAccess: true`; current public containers use anonymous blob access.
- Public containers currently configured in `AzureContainerPropertiesMap`: `AppAssets`, `DungeonsAssets`, and `PublicUserAssets`.
- `AppAssets` is used for public app assets such as `logo.png`.
- `DungeonsAssets` is used by the app's dungeons asset routes and the Phaser file-pack script.
- `PublicUserAssets` is used by user asset upload/update flows.
- Development storage has one explicit IP rule and `defaultAction: Allow`; production storage has no explicit IP rules and `defaultAction: Allow`.
- No storage v2 changes should be applied until recovery, retention, and public asset requirements are chosen.
