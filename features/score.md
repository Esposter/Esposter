# Esposter — Repository Score

**Overall: 84 / 100**

A well-engineered, TypeScript-strict monorepo with consistent architectural patterns and comprehensive linting. The main drag is beta/RC dependencies in production paths and uneven test coverage at the composable layer.

---

## Architecture & Code Organisation — 20 / 20

**Strengths**

- Clean package separation: `app`, `shared`, `db`, `db-schema`, `azure-functions`, `vue-phaserjs` all have distinct responsibilities with a sensible dependency DAG (shared has no Vue deps, db-schema has no server deps, etc.)
- Consistent data-driven map pattern throughout: `*TypeColorMap`, `*TypeCommandMap`, `*TypeCreateMap`, `ColumnStatDefinitionMap`, `ColumnChartDataMap`, `ColumnTransformationComputeMap` — new variants require adding one entry, not modifying logic
- Command pattern for undo/redo is clean and well-scoped: each command encapsulates `doExecute` / `doUndo` against the item, history stack is separate concern
- One export per file, no `export {}` syntax, barrel files generated by ctix — no accidental re-export drift
- File naming follows the convention (camelCase functions, PascalCase constants/models) consistently

---

## TypeScript — 15 / 15

**Strengths**

- `strict: true`, `noUncheckedIndexedAccess: true`, `noImplicitOverride: true`, `isolatedModules: true`, `isolatedDeclarations: true`, `verbatimModuleSyntax: true` — one of the strictest configs you can have
- `noUncheckedIndexedAccess` is actively respected: `takeOne()` from `@esposter/shared` used in place of direct index access across the codebase
- Zod schemas co-located with models, `satisfies z.ZodType<T>` pattern enforced, no raw `any` or `Omit`
- Enum maps over string unions, `as const satisfies` for constant objects

**Remaining notes**

- `skipLibCheck: true` (acceptable for build speed, but hides type errors in external packages)

---

## Code Quality & Patterns — 15 / 15

**Strengths**

- Guard clauses over nested if/else throughout TypeScript files
- `InvalidOperationError` for impossible states instead of silent fallbacks
- `structuredClone(toRawDeep(...))` snapshot + restore pattern for optimistic updates is consistent and correct
- vjsf + Immer plain-object binding pattern documented and followed (class instances never passed directly to `<Vjsf>`)
- Zod form schemas separated from entity schemas (`*FormSchema` via `.pick().extend()`) prevents vjsf from seeing `z.date()` fields

---

## Testing — 8 / 10

**Strengths**

- 184 test files across the monorepo; commands, pure services, and shared utilities are well covered
- 7 benchmark files for performance-critical table editor commands (`CreateRowsCommand`, `DeleteColumnsCommand`, etc.)
- Canonical test value conventions enforced (no "Alice"/"Bob", no "non-existent-id")
- PGlite (in-memory PostgreSQL) used for server-side tests — no real DB required

**Gaps**

- `packages/azure-functions` has no visible test files — webhook handlers are untested
- Pinia stores largely untested at the unit level (acceptable if composable tests cover the call chain, but composable tests are sparse too)

---

## Security — 8 / 10

**Strengths**

- Zod `.safeParse()` on all tRPC inputs, route params, and webhook handlers — no unvalidated user data reaches business logic
- Rate limiting via `rate-limiter-flexible` with IP + user keying, proper `429 + Retry-After`, bypassed only in non-production
- `better-auth` for session management with OAuth (FB, GitHub, Google) and Drizzle adapter
- Array bounds enforced on bulk operations (`.min(1).max(MAX_READ_LIMIT)`)
- Drizzle ORM parameterized queries prevent SQL injection

**Gaps / Accepted Trade-offs**

- `unsafe-eval` in CSP required by Desmos graphing library — unavoidable, documented
- `unsafe-inline` required by Vuetify style injection and Nuxt hydration — unavoidable
- `xssValidator: false` due to tRPC-Nuxt issue #215 — track resolution upstream
- No CSRF protection configured — `nuxt-security` v2 requires explicit `csrf: { enabled: true }` opt-in (not set in `configuration/security.ts`); tRPC's JSON content-type blocks HTML-form CSRF but not fetch-based attacks if CORS is permissive

---

## Dependencies — 6 / 10

**Healthy**

- Catalog-driven versioning in `pnpm-workspace.yaml` with `catalogMode: strict` — no version drift between packages
- Core tools pinned: `nuxt: 4.3.1`, `typescript: 5.9.3`, `ctix: 2.7.2`
- Zod 4.x, Drizzle 0.45.x, better-auth 1.5.x — all recent stable releases

**Risks (external, hard to control)**

- `phaser: ^4.0.0-rc.7` — RC; game engine is a core feature. Breaking changes possible before 4.0 stable
- `vuetify-nuxt-module: ^1.0.0-beta.2` — beta; primary UI integration layer. Has already required workarounds (layer order bug, disableVuetifyStyles)
- `survey-core / survey-creator-core / survey-creator-vue / survey-vue3-ui: 3.0.0-beta.0` — 4 beta packages for the survey feature
- `rolldown: ^1.0.0-rc.12` — RC bundler in the build chain; blocks all package builds if it regresses
- `@typescript/native-preview: ^7.0.0-dev.20260330.1` — dev preview; used for typecheck speed, not correctness

The `eslint-plugin-depend` is configured and will surface issues here over time.

---

## CI / CD — 8 / 10

**Strengths**

- Full pipeline on every push: install → build → lint → format check → typecheck → test
- Lerna-lite orchestrates multi-package builds with correct dependency order
- `oxlint` + `oxfmt` (Rust-based) for fast feedback; `eslint` for deeper type-aware rules
- Separate `eslint.light.config.js` for fast lint:fix vs full lint:check

**Gaps**

- No parallelisation between lint, typecheck, and test jobs — they run sequentially
- No coverage thresholds enforced (reporting added via `pnpm coverage` + artifact upload)

---

## Bundle & Performance — 6 / 10

**Strengths**

- Selective transpilation (heavy PDF libraries only on server)
- `@vue-pdf-viewer` excluded from Vite pre-optimization (lazy-loaded)
- `assetsInlineLimit: 0` prevents Phaser from breaking on inline data URIs
- Benchmark infrastructure in place for table editor hot paths

**Concerns**

- Very large dependency footprint: Phaser, GrapesJS, Survey (4 packages), Three.js, FullCalendar, pdf-viewer — all potentially included in the main bundle
- No explicit route-level or component-level lazy loading visible in config
- `pnpm analyze` exists but no evidence it is run regularly or has size budgets
- Rolldown RC bundler: optimisation behaviour may differ from Vite in non-obvious ways until stable
